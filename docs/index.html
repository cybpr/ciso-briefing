Linux "Copy Fail" CVE-2026-31431 — Universal Kernel Privilege Escalation & Container Escape

Source: Xint.io / Theori Research  |  Published: April 29, 2026  |  CVSS: 7.8 HIGH
9.2

S:9 × E:10 × X:8
MITRE ATT&CK:   T1068 — Exploitation for Privilege Escalation T1611 — Escape to Host (Container) T1055 — Process Injection

Cybersecurity researchers have disclosed details of a Linux local privilege escalation (LPE) flaw that could allow an unprivileged local user to obtain root. The high-severity vulnerability tracked as CVE-2026-31431 (CVSS score: 7.8) has been codenamed "Copy Fail" by Xint.io and Theori. At its core, the vulnerability stems from a logic flaw in the Linux kernel's cryptographic subsystem, specifically within the algif_aead module — an issue introduced in a source code commit made in August 2017. Successful exploitation could allow a simple 732-byte Python script to edit a setuid binary and obtain root on essentially all Linux distributions shipped since 2017, including Amazon Linux, RHEL, SUSE, and Ubuntu. Critically, the exploit modifies the in-memory version of a file (the kernel's page cache) but never the on-disk version — meaning file hash-based integrity monitoring tools will see absolutely nothing. Copy Fail is not just a local privilege escalation — it is a container escape primitive and a Kubernetes node compromise vector. A working public PoC is available and patches are shipping from major distributions now. Exploitation requires local code execution as a regular user, meaning it cannot be exploited remotely on its own — but chain it with anything that gives you that (web RCE landing in an unprivileged service account, an SSH foothold, a malicious PR on a CI runner) and you're root.

⚡ Immediate Action: Patch all Linux kernels to versions ≥ 6.18.22 / 6.19.12. Blacklist the algif_aead module as a temporary mitigation where kernel updates cannot be applied immediately. Note: modprobe-based workarounds do not work on RHEL-family distributions (CONFIG_CRYPTO_USER_API_AEAD=y is built-in). Deploy runtime detection rules for anomalous AF_ALG AEAD socket creation (MITRE TA0004).

GitHub Enterprise Server RCE — CVE-2026-3854 (88% of Self-Hosted Instances Unpatched)

Source: Wiz Research  |  Published: April 28, 2026  |  CVSS: 8.7 HIGH
8.7

S:9 × E:9 × X:8
MITRE ATT&CK:   T1190 — Exploit Public-Facing Application T1059 — Command and Scripting Interpreter T1552 — Unsecured Credentials

CVE-2026-3854 is a critical flaw in GitHub's internal git infrastructure enabling RCE on GitHub.com and GitHub Enterprise Server. Discovered by Wiz Research, the flaw allows any authenticated user to execute arbitrary commands on GitHub's backend servers with a single git push command — using nothing but a standard git client. During a git push operation, user-supplied push option values were not properly sanitized before being included in internal service headers; because the internal header format used a delimiter character that could also appear in user input, an attacker could inject additional metadata fields through crafted push option values. GitHub.com was patched within two hours of disclosure on March 4, 2026. However, as of public disclosure on April 28, Wiz data indicates that 88% of GitHub Enterprise Server instances are still vulnerable. Notably, this is one of the first critical vulnerabilities discovered in closed-source binaries using AI, highlighting a shift in how these flaws are identified. GitHub conducted a forensic investigation and found that all instances of the anomalous internal code path were linked exclusively to the Wiz researchers' own testing — providing strong assurance that the vulnerability had not been abused before disclosure.

⚡ Immediate Action: Upgrade all GitHub Enterprise Server instances to version 3.19.3 or later immediately. Review /var/log/github-audit.log for push operations containing semicolons (;) in push options to audit for pre-patch exploitation. GitHub.com and GitHub Enterprise Cloud users require no action.

VECT 2.0 — RaaS Wiper Masquerading as Ransomware Targeting Windows, Linux & ESXi (No Recovery Possible)

Source: Check Point Research  |  Published: April 28, 2026

S:10 × E:7 × X:8
MITRE ATT&CK:   T1485 — Data Destruction T1486 — Data Encrypted for Impact T1195 — Supply Chain Compromise T1021.002 — SMB/Windows Admin Shares

Threat hunters are warning that the cybercriminal operation known as VECT 2.0 acts more like a wiper than ransomware due to a critical flaw in its encryption implementation across Windows, Linux, and ESXi variants that renders recovery impossible even for the threat actors. A critical flaw in the encryption implementation — identical across all three platform variants — discards three of four decryption nonces for every file above 131,072 bytes (128 KB). Full recovery is impossible for anyone, including the attacker. At a threshold of only 128 KB, this effectively makes VECT a wiper for virtually any file containing meaningful data, including enterprise assets such as VM disks, databases, documents, and backups. VECT operators announced a partnership with TeamPCP, the threat group responsible for recent supply-chain attacks impacting Trivy, LiteLLM, and Telnyx, as well as an attack against the European Commission — with the stated goal of exploiting victims of those supply-chain compromises by deploying ransomware payloads in their environments. In a further escalation, VECT announced a partnership with BreachForums itself, granting every registered forum member automatic affiliate status — access to the ransomware builder, negotiation platform, and leak site with no entry fee required.

⚡ Immediate Action: Do NOT pay any VECT ransom — recovery is mathematically impossible for files over 128KB. Validate offline/immutable backup integrity now, especially for ESXi VM disks and databases. Deploy IOC-based detection: monitor for PowerShell disabling of Windows Defender, event log clearing, and unusual Safe Mode boot config changes. Search VECT's known file extension (.vect) and ransom note (!!!READ_ME!!!.txt) across endpoints.
